# Findings report — host-side fio characterisation of the `data` pool
**Period covered:** 2026-04-21 → 2026-04-25
**Hosts measured:** m-p-proxmox-08 (initial), m-p-proxmox-05 (parity)
**Status of analysis:** snapshot in time; ARC-size sweep in progress on both hosts

---

## 1. Executive summary

Five takeaways, ordered by importance.

1. **The pool's read cache (ARC) is capped at 8 GiB on both nodes** — confirmed
   from `/sys/module/zfs/parameters/zfs_arc_max` snapshots. Working sets in
   every test we ran are 25–60× larger than ARC, so the IOPS and latency
   numbers we measured are real-disk numbers, not RAM. This is the single most
   important fact for interpreting everything below.
2. **Stock-Proxmox SQL Server-shaped workload fragments the pool fast.** A 26 h
   sustained run drove pool fragmentation from 19 % → 37 % (m-p-proxmox-08) and
   5 % → 31 % (m-p-proxmox-05) while pool capacity barely moved. With autotrim
   off and COW on, write churn against the same regions scatters free space
   regardless of how much capacity is in use.
3. **Fragmentation cost is latency-tail-shaped, not throughput-shaped.** Mean
   IOPS hold up under fragmentation; per-I/O p99 / p99.9 sync-write latency
   does not. On m-p-proxmox-05 the 26 h run measured a real per-I/O p99 of
   **35.9 ms** for SQL log commits and a p99.9 of **64.7 ms** — that is the
   number a SQL Server SLO actually feels.
4. **The zvol path is materially more robust than the dataset path** for SQL
   Server-shaped workloads. On a fresh pool the zvol variant was ~2× better on
   write tail latency; on a fragmented pool the gap widens to ~4–5×.
5. **The two hosts produce meaningfully different baseline numbers because
   the hardware is materially different**, despite a stated "aligned and
   comparable" configuration. -08 has Samsung PM9A3 NVMe drives in its data
   pool; -05 has Micron 7400-series. Different vendors, different controller
   firmware, different performance envelopes. The two hosts also run
   different OpenZFS kernel module versions (2.4.0 on -08 vs 2.3.4 on -05).
   Section 6 documents the env-diff that surfaced this. Cross-host numbers
   are therefore *not* directly subtractable; any "VM stack tax" measurement
   must compare host-side and in-VM runs on the *same* host.

## 3. Methodology

The framework (this repository) implements a strict separation between
**production-representative** tests (untouched datasets and a stock zvol —
nothing tuned) and **diagnostic** tests (one ZFS property deliberately
non-default, used to explain *why* a stock setting behaves a certain way).
Production-representative numbers describe what a real Proxmox VM on this
pool would actually experience; diagnostic numbers explain the boundaries.

For each fio invocation the framework captures, in a per-run timestamped
result directory:
- `fio.json` — clean parseable JSON (post-extracted from fio's combined output)
- `fio.summary.txt` — human-readable per-job summary
- `effective.fio` — the exact job file that ran (global prelude + selected job)
- `env/` — system, ZFS, ZFS tunables, storage controllers, fio version,
  Defender/services on Windows, tool availability
- `monitor/` — `zpool-iostat.log`, `vmstat.log`, `iostat.log`, `mpstat.log`,
  ARC counters, `MONITORS_SUMMARY.txt` listing what ran and what was skipped
- `zpool-before.txt`, `zpool-after.txt` — pool capacity / fragmentation deltas

A 26 h SQL Server-shaped endurance test produced separate per-stream
bw/iops/lat time-series logs, allowing the four-hour-window analysis used in
section 4.

## 4. Single-host characterisation (m-p-proxmox-08)

The original suite (host suite + 26 h endurance) ran on m-p-proxmox-08
between 2026-04-22 and 2026-04-23. Five concurrent multi-stream workloads
were re-run on 2026-04-24 against the now-fragmented pool to recover
per-stream JSON visibility (lost in the original due to a fio
`group_reporting` configuration issue, since fixed).

### 4.1 Sequential and random baseline (fresh pool, ARC=8 GiB)

| Job | r IOPS | w IOPS | r p99 | w p99 | w p99.9 |
|---|---:|---:|---:|---:|---:|
| `seq-read-1m`     | 27,745 | —      | 25.6 ms | —       | —        |
| `seq-write-1m`    | —      | 5,888  | —       | 217 ms  | 321 ms   |
| `rand-read-4k`    | 232,045| —      | 3.4 ms  | —       | —        |

Reproducibility: each baseline ran twice. Reads delta <0.5 %; writes delta

### 4.2 Workloads — production-representative vs comparison

| Run | r IOPS | w IOPS | r p99 | w p99 | w p99.9 |
|---|---:|---:|---:|---:|---:|
| `sqlserver-zvol`               | 39,732 | 21,614 | 1.9 ms | 2.3 ms | 4.2 ms |
| `sqlserver-sim` (dataset)      | 20,808 | 11,369 | 1.8 ms | 7.5 ms | 9.2 ms |
| `sqlserver-zvol-storm`         | 37,565 | 19,527 | 4.8 ms | 6.5 ms | 9.1 ms |
| `sqlserver-storm` (dataset)    | 28,190 | 12,626 | 2.0 ms | **25.8 ms** | 29.8 ms |
| `oltp-16k`                     | 34,541 | 14,786 | 2.0 ms | 13.6 ms | 18.0 ms |
| `vm-mixed`                     | 21,946 | 11,082 | 3.9 ms | 12.5 ms | 16.1 ms |
| `backup-ingest` (1M sequential)| —      | 2,939  | —      | 60.6 ms | 69.7 ms |

**Two findings stand out.**

The zvol path is *uniformly* better than the dataset path for SQL
Server-shaped workloads: ~2× throughput, ~3× lower write tail latency. On the
checkpoint-storm scenario (sustained OLTP + sync log + delayed write burst)
the dataset variant produced p99 = 25.8 ms; the zvol variant produced 6.5 ms.
This is the primary reason Proxmox defaults VM disks to zvols.

Backup-ingest (1 MiB sequential incompressible writes from many streams) is
limited by the LUKS encrypt CPU plus the txg flush rhythm: 2.9 GB/s sustained
with ~60 ms p99 latency tails. Per-vdev `zpool iostat` showed all three
mirrors carrying comparable load — RAID10 balancing is healthy.

### 4.3 26-hour SQL Server endurance — the headline finding

`jobs/endurance/sqlserver-zvol-26h.fio` runs the same three concurrent
streams as `sqlserver-zvol.fio` (data 200 GiB, tempdb 40 GiB, log 20 GiB) at
runtime=93 600 s. Per-stream IOPS over 4 h windows, computed from the
per-second time-series logs:

| Window | sql-data r IOPS | sql-data w IOPS | sql-log mean clat |
|---|---:|---:|---:|
| 0–4 h    | 10,360 | 4,439 | 2,054 µs |
| 4–8 h    | 9,191  | 3,939 | 2,188 µs |
| 8–12 h   | 8,811  | 3,777 | 2,576 µs |
| 12–16 h  | 8,619  | 3,693 | 2,912 µs |
| 16–20 h  | 8,570  | 3,673 | 3,160 µs |
| 20–24 h  | 8,547  | 3,663 | 3,330 µs |
| 24–28 h  | 8,480  | 3,634 | 3,295 µs |

**Monotonic 18 % IOPS decline, 60 % rise in mean log commit latency**, while
pool fragmentation went 19 % → 37 % and capacity moved from 6 % → 7 %. All
three mirror vdevs fragmented identically.

### 4.4 Diagnostic explanations of stock behaviour

| Diagnostic | What it tells us about the stock pool |
|---|---|
| `recordsize-aligned-vs-misaligned` | 16K random writes on rs=128K (stock) hit 17.9 K IOPS p99=30.8 ms vs aligned rs=16K 44.4 K IOPS p99=9.8 ms. **2.5× IOPS gap, 3× latency gap** is why Proxmox uses zvols (volblocksize=16K) for VMs rather than file-on-dataset. |
| `arc-bypass` (primarycache=metadata) | rand-read-4k IOPS identical to default — confirming that default reads were NOT being served from ARC (working set far exceeds 8 GiB cap). |
| `compressible-vs-incompressible` | Zero-fill writes hit 293 K IOPS / 36 GB/s; incompressible random 19 K IOPS / 2.4 GB/s. lz4 swallows zero pages entirely. **Any benchmark not using `refill_buffers=1 buffer_compress_percentage=0` is measuring RAM, not disk.** |
| `compression-off` | Incompressible-random write throughput 14 % higher with compression off vs lz4. lz4 short-circuits on random data quickly enough that turning it off saves CPU but only marginally. Stock lz4 is right. |
| `sync-zil-hammer` | sync=disabled 20.5 K IOPS / p99 1.4 ms; sync=standard (stock) 16.8 K / 1.3 ms; sync=always 16.0 K / 1.5 ms. **In-pool ZIL costs ~18 % throughput and almost nothing in tail latency.** Adding a SLOG would not buy meaningful improvement on this NVMe stack. |
| `ioengine-matrix` | io_uring delivers 1.8× more IOPS than libaio/sync but its write p99.9 was 33 ms vs sync 1.7 ms. Proxmox 7.2+ qemu defaults to aio=io_uring, so VMs get this profile — throughput win, tail-latency variance under load. |
| `qd-sweep` | Sweet spot QD=16–64 for single-thread random read. Past QD=64 latency climbs faster than IOPS. QD=256 hits 230 K IOPS @ 2.7 ms p99 — a clear cliff. |
| `numjobs-sweep` | Sweet spot 4–16 concurrent writers. Past 16 you queue without throughput gain; 64 jobs hit 60 ms p99. |
| `access-latency` | 4K random read QD=1 floor: 177 µs p50, 216 µs p99. Single-thread cold-data ("seek") latency on this LUKS+NVMe stack. |

### 4.5 Refined post-26 h short-workload re-runs

The five concurrent-multi-stream workloads were re-run against the now-39 %
fragmented pool. Per-stream JSON was now extractable (framework fix in place).
Aggregate IOPS were equal-or-better than on the fresh pool — but per-IO p99.9
write latencies on dataset-backed sync streams worsened sharply:

- `sqlserver-sim/sql-log` p99.9 hit **130 ms** on the fragmented pool. The
  zvol equivalent stayed at single-digit ms.
- `sqlserver-storm/checkpoint-burst` (dataset) p99 = 33.8 ms vs zvol variant
  7.4 ms — **4.5× wider gap on fragmented pool than on fresh pool (which was
  already 2×)**.

The dataset-vs-zvol gap *widens under fragmentation*. This confirms the
zvol-first stock Proxmox design holds up under pool aging.

## 6. Cross-host divergence — root cause identified

A diff of the per-run env snapshots (`env/zfs.txt`, `env/zfs-tunables.txt`,
`env/storage.txt`) between the two hosts produced three concrete differences
that together explain the divergence. The hypothesis matrix below is
preserved, but H2 (NVMe firmware/model) and H10 (aligned-but-not-identical
configuration) are now confirmed as the dominant causes.

### 6.0 The three differences

**Different NVMe drives.** The data pools on the two hosts use *different
drive vendors and models*:

| Host | Data-pool NVMe model |
|---|---|
| m-p-proxmox-08 | 6× **SAMSUNG MZQL215THBLA-00A07** (Samsung PM9A3 series, 15 TB enterprise NVMe) |
| m-p-proxmox-05 | 6× **MTFDKCC15T3TGP-1BK1DABYY** (Micron 7400/7500 series, 15.3 TB enterprise NVMe) |

Samsung PM9A3 and Micron 7400-series drives have different controller
firmware, different SLC-cache strategies, different garbage-collection
behaviour, and different read/write performance envelopes. They are not
interchangeable in any meaningful sense. The "configurations are aligned and
completely comparable" assumption was incorrect; the hosts share OS-level
software but their storage hardware is fundamentally different.

This single fact explains the asymmetric performance pattern in section 5.1:
the Samsung drives in -08 are evidently faster on small random reads and
sequential reads, while the Micron drives in -05 are faster on small random
writes and have lower sequential-write tail latency. This is consistent with
how the two product lines are typically benchmarked.

**Different ZFS version on the two hosts.**
- m-p-proxmox-08: `zfs-2.4.0-pve1` userspace + `zfs-kmod-2.4.0-pve1` kernel
  module (matched)
- m-p-proxmox-05: `zfs-2.4.1-pve1` userspace + `zfs-kmod-2.3.4-pve1` kernel
  module (**mismatched**; userspace is 2.4.1 but the running kernel module
  is the older 2.3.4)

The kernel module is the version that actually executes ZFS code in the I/O
path. -05 is running OpenZFS 2.3.4 internally despite reporting 2.4.1 from
userspace. ZFS 2.3 → 2.4 includes notable changes to the ARC eviction path
and the direct-I/O property; some of the read-side performance gap may stem
from this.

**Different recent pool history on -05.** From `env/zfs.txt`:
> `scan: resilvered 96K in 00:00:00 with 0 errors on Fri Apr 17 10:05:42 2026`

-05 had a resilver finish a week before our tests began, meaning at least
one drive in -05 was replaced or rebuilt recently. -08 had only a clean
scrub on April 12 (no resilver). A recently-resilvered drive may have
different fragmentation / wear-leveling state than its peers.

**Other observations from the env diff:**
- ARC counters at run start differ (-08 had 447 M hits accumulated from
  prior tests; -05 had 34 M — a much fresher ARC). This is a downstream
  effect of test-run timing, not an independent cause.
- Both hosts have OS disks from yet a third vendor (-08: Micron 7450 OS
  disks; -05: Kioxia KCD8 OS disks). Doesn't affect the data pool but
  illustrates the inhomogeneity.

### 6.1 What we know for certain

- Both hosts run Proxmox 9.1.6, kernel 6.17.9-1-pve, ZFS 2.4.0, qemu 10.1.2.
  Verified via `pveversion -v` and the per-run `env/system.txt`.
- Both hosts have ARC max set to 8 GiB.
- Both hosts have the same CPU model (Xeon Gold 5412U).
- Both pools are RAID10 of 6× LUKS-encrypted 14 TB NVMe at ashift=12.
- The framework code, fio version (3.39), and job files are identical.
- Pool starting fragmentation differed: -08 was 19 %, -05 was 5 %.
- ARC residency at run start differed: -08 had 2.7 GiB populated (priors
  from earlier tests), -05 had 8.3 GiB populated (full).

### 6.2 Hypothesis matrix

| Hypothesis | Direction it would push | Supported by | Refuted by | Verdict |
|---|---|---|---|---|
| **H1: Other VMs running on -05 during the test** | -05 worse on reads (cache contention) but mixed on writes | -05 is the host carrying VM 110 (the Windows IO test VM); we did not verify the VM was off during the host suite run | We have no direct evidence of running guests during the test | **Likely contributor; needs verification** — query `qm list` history or systemd journal for VM activity in the test window |

So if anything, this hypothesis predicts the wrong direction for reads. | **Refuted as primary cause; possible secondary effect** |
| **H4: ARC residency at start (-05 ARC was full, -08 was 1/3 full)** | -05 should be *faster* on early reads (warm cache hits) — opposite of what we measured | -05 had 8 GiB of cached data at start | If that 8 GiB matched what fio re-reads, -05 wins. **Check B — search the -05 test window for concurrent activity.**
Look at `monitor/zpool-iostat.log` and `monitor/vmstat.log` for the -05 runs
around the times of the slow baselines. Any non-test I/O (scrub, snapshot
replication, VM activity) will show up as activity outside the fio profile.

### 6.4 Experiments needed

The ARC sweep currently in motion will tell us how each host responds to
ARC sizing, which is useful in its own right but does not directly resolve
the cross-host divergence. Three additional, targeted experiments will:

1. **Quiet-host baseline pair.** On both -05 and -08, with all VMs powered
   off and no scheduled jobs, run only `bin/run-suite.sh baseline` three times
   in a row. If the gap closes when the host is otherwise idle, H1 is
   confirmed. If it persists, H2 / H5 / H10 remain.
2. **NVMe identity audit.** Install `nvme-cli` on -08, capture
   `nvme list` + `nvme id-ctrl /dev/nvmeXn1` per drive, plus
   `smartctl -a /dev/nvmeXn1`, on both hosts. Compare firmware revs, model
   numbers, percentage-used / wear-leveling counts, temperature history.
3. **Run-to-run variance characterisation.** Run `bin/run-suite.sh baseline`
   ten times in a row on a single host (say -05) with no other changes.
   Establish the actual variance envelope so future "delta" claims are
   compared against a real noise floor, not the 8 % we estimated from a
   sample of two.

### 6.5 Until that data is in

We cannot reliably attribute the host-vs-VM "Proxmox stack tax" by
subtracting -05 numbers from -08 numbers. Any in-VM Windows results must be
compared against the -05 host runs (same physical hardware). The Sunday
plan to run the host suite on -05 alongside the VM suite was correct; it
gives us the only valid comparison.

## 7. ARC sweep findings (2026-04-27)

`bin/sweep-arc.sh` ran on both hosts on 2026-04-27 with sizes 8 / 32 / 64 /
128 / 256 GiB and `FIO_RUNTIME=120` s per stage, executing
`scaling/access-latency.fio` + `workloads/sqlserver-zvol.fio` at each
size. Both hosts had had their default `arc_max` raised since the
original test runs (-08: 80 GiB original, -05: 377 GiB original) — the
sweep restored these on exit via the trap.

### 7.1 Access-latency floor is hardware, not cache

Single-thread random 4K read latency at QD=1 was effectively constant
across ARC sizes on both hosts:

| Host | ARC | cold-4k-qd1 IOPS | p50 µs | p99 µs |
|---|---:|---:|---:|---:|
| -08 (Samsung PM9A3) | 8–256 GiB | ~5,500 | **177** | ~218 |
| -05 (Micron 7400)   | 8–256 GiB | ~2,850 | **367** | ~424 |

**The Samsung NVMe stack delivers single-thread random reads at twice
the rate of the Micron stack.** ARC size is irrelevant at this
single-I/O-per-thread regime because the 120 s runtime never reaches a
steady-state cache-hit regime against the 50 GiB working file. This
quantifies one of the two main hardware-vendor differences directly.

### 7.2 The IOPS-decreases-with-ARC anomaly

Both hosts show monotonically *declining* sql-data-zvol IOPS as ARC
grows from 8 GiB → 256 GiB:

| Host | ARC | sql-data r IOPS | sql-data w IOPS | r p99 | w p99 |
|---|---:|---:|---:|---:|---:|
| -08 | 8   | **104,278** | **44,652** | 1.7 ms | 1.7 ms |
| -08 | 32  | 72,725  | 31,190 | 1.9 ms | 1.9 ms |
| -08 | 64  | 55,038  | 23,578 | 2.1 ms | 2.1 ms |
| -08 | 128 | 50,023  | 21,457 | 2.1 ms | 2.2 ms |
| -08 | 256 | 43,610  | 18,688 | 2.2 ms | 2.3 ms |
| -05 | 8   | **97,339** | **41,755** | 1.2 ms | 1.8 ms |
| -05 | 32  | 80,371  | 34,438 | 1.1 ms | 2.1 ms |
| -05 | 64  | 76,387  | 32,714 | 0.9 ms | 2.2 ms |
| -05 | 128 | 70,446  | 30,190 | 0.9 ms | 2.3 ms |
| -05 | 256 | 68,046  | 29,148 | 1.0 ms | 2.5 ms |

**This is not a cache benefit reversal.** Most likely explanation: at
120 s runtime, ARC never converges. The sweep script drops ARC to 1 GiB
between stages then sets the new ceiling, so larger ceilings spend more
of the 120 s "still ramping". The kernel pays memory-management cost
(allocation, balance, slab churn) without delivering the cache-hit
benefit of a populated cache. The remedy is **longer runtime** —
`FIO_RUNTIME=1800` (30 min) per stage would let ARC populate against
the 200 GiB working set before stats collection covers steady state.

### 7.3 sql-log latency: hosts diverge in opposite directions

The most operationally important finding. Sync 60K log writes (the SQL
transaction-commit path):

| Host | ARC | w IOPS | w p99 ms | w p99.9 ms |
|---|---:|---:|---:|---:|
| -08 | 8   | 100 | 60.6 | 93.8 |
| -08 | **32**  | **143** | **18.5** | **25.3** ← best |
| -08 | 64  | 69  | 94.9 | 135.3 |
| -08 | 128 | 95  | 128.5 | 244.3 |
| -08 | 256 | 78  | **160.4** | **308.3** ← worst |
| -05 | 8   | 81  | 42.7 | 65.3 |
| -05 | 32  | 125 | 32.6 | 56.9 |
| -05 | 64  | 160 | 23.7 | 43.8 |
| -05 | 128 | 178 | 20.3 | 36.4 |
| -05 | 256 | **188** | **18.2** | **33.8** ← best |

- **-08 (Samsung): clear sweet spot at ARC=32 GiB.** Goes 9× *worse* on
  log p99 from ARC=32 to ARC=256.
- **-05 (Micron): monotonic improvement.** Best p99 at ARC=256.
- **Both converge at ~18 ms p99** — but at *very different ARC sizes*
  (32 vs 256 GiB). Setting a uniform `zfs_arc_max` across the cluster
  produces the best result on one host and a near-pathological result
  on the other.

### 7.4 Dirty-data flush hypothesis

A specific candidate for the -08 sql-log regression: `zfs_dirty_data_max`
defaults to 10 % of physical memory (capped at `zfs_dirty_data_max_max`,
default 25 %). When ARC max grows, the dirty-data buffer typically
grows with it — and on the bigger ceilings that means ZFS lets a much
larger window of dirty data accumulate before forcing a txg flush. Each
flush is then bigger and burstier. The Samsung NVMe in -08 appears to
respond worse to those bursts than the Micron in -05 — it produces
multi-hundred-millisecond stalls in the sync log stream while txg-flush
write storms are draining. The Micron drives apparently smooth out the
flush traffic and finish it without crowding the sync stream.

This is testable: cap `zfs_dirty_data_max` to a fixed small value (e.g.
4 GiB) on -08 with ARC=256 and re-run. If sql-log p99 drops back into
the tens of ms, the hypothesis is confirmed: the issue is dirty-buffer
scaling, not ARC itself, and -08 can run with ARC=256 *and* a capped
dirty buffer for a per-host configuration that satisfies both reads and
log latency.

`bin/sweep-dirty-data.sh` automates exactly this experiment — same
shape as `sweep-arc.sh`, sweeping `zfs_dirty_data_max` across a series
of values with the same job set, trap-protected restore on exit.

### 7.5 Caveats — read these before tuning anything

1. **120 s runtime is too short** for ARC to populate to large
   ceilings. Re-run the sweep with `FIO_RUNTIME=1800` (30 min/stage)
   to amortise ARC fill cost. This is the single most important
   follow-up before drawing tuning conclusions.
2. **Single run per stage.** No variance bounds. A 308 ms p99.9 from a
   single 120 s window could be one bad outlier. Reproducibility needs
   ≥3 runs per stage at the same setting.
3. **Eviction-then-grow protocol.** `sweep-arc.sh` drops ARC to 1 GiB
   between stages so each stage starts cold-ish. This is what makes
   short runs unfair to large ARC sizes — the larger the target, the
   more of the 120 s is spent ramping. A protocol that pre-warms ARC
   for, say, 5 minutes before the timed measurement would change the
   picture; we have not run it that way.
4. **No long-runtime data yet.** Whether the IOPS-decreases-with-ARC
   pattern persists or inverts at long runtime is the open question.
   Until that's known, do not change cluster ARC settings based on
   this sweep alone.

## 8. Framework caveats and corrections already applied

This section captures the framework changes already made and the run-time
practices we'd insist on next time.

- `--output-format=json+,normal` writes both formats to one file → un-parseable
  as JSON. **Fixed**: `bin/run-suite.sh` now writes the combined output to
  `fio.out` and post-extracts a clean `fio.json` plus a human-readable
  `fio.summary.txt`. Old result dirs require manual JSON extraction.
- `group_reporting=1` in `_global.fio` collapsed concurrent multi-stream
  jobs into one aggregated row. **Fixed**: `new_group=1` added to each
  stream in the six concurrent multi-stream files. Original 26 h run on -08
  has only the per-second time-series logs to recover per-stream stats from.
- Monitor tools were not asserted at startup → silent skips on
  m-p-proxmox-08 (no sysstat → no iostat/mpstat data; no arcstat → kstat
  fallback). **Fixed**: `bin/monitor.sh` now writes `MONITORS_SUMMARY.txt`,
  fails fast on missing critical tools, and `apt install fio sysstat
  zfsutils-linux nvme-cli cryptsetup procps` is now in README prerequisites.
- The metadata-heavy job hit `RLIMIT_NOFILE=1024` because thread=1 jobs
  share an FD table. **Fixed**: `bin/run-suite.sh` raises the soft limit
  to the hard cap; the job file uses a more conservative `openfiles=64`.
- Running the suite without explicitly recording the pool's starting state
  makes "fragmentation cost" hard to attribute. The framework now snapshots
  `zpool list -v` before and after every job, but a clean-state baseline
  (`bin/cleanup.sh && bin/prepare-datasets.sh && bin/prepare-zvols.sh`)
  before every endurance run is the only way to get truly comparable
  fragmentation deltas.

## 9. Recommended next steps

In priority order, refreshed after the ARC sweep:

1. **Re-run the ARC sweep with `FIO_RUNTIME=1800`** on both hosts. The
   single most important follow-up. The 120 s sweep we have shows monotonic
   IOPS *decline* with larger ARC, almost certainly because the runtime is
   too short for ARC to populate. Until we have long-runtime data we cannot
   make tuning decisions.
2. **Run `bin/sweep-dirty-data.sh`** on both hosts (added 2026-04-27). Tests
   whether the -08 sql-log regression at large ARC is driven by
   `zfs_dirty_data_max` scaling rather than ARC itself. If confirmed, -08
   can run at ARC=256 with `zfs_dirty_data_max=4G` and avoid the 308 ms
   p99.9 — a genuinely useful operational finding.
3. **Reconcile ZFS module versions across the cluster.** -05 has a
   userspace/kmod mismatch (userspace 2.4.1, kmod 2.3.4) — almost certainly
   a node that was upgraded but not rebooted. Reboot to load the new
   module, or document the divergence and re-run after reboot. Until
   reconciled, the cluster cannot claim to be "aligned" at the ZFS layer.
4. **NVMe identity audit on -08.** Install `nvme-cli` (it is missing from
   -08's env snapshots) and capture `nvme list` + `nvme id-ctrl` +
   `smartctl -a` per drive on both hosts. Compare firmware revs,
   percentage-used / wear-leveling counts, thermal history. We now know
   the *models* differ; we still want to know how worn each drive is and
   at which firmware revision.
5. **In-VM Windows runs on -05** with the existing `win/` framework, paired
   against the -05 host runs (not -08). Three layers will then be
   measurable on identical hardware: host on zvol → VM on NTFS → SQL
   Server itself.
6. **Run-to-run variance characterisation.** Run `bin/run-suite.sh baseline`
   five times in a row on a single host with no other changes. Establish
   the actual noise floor so future delta claims are compared against a
   real number, not the 8% we estimated from a sample of two.
7. **Re-run a 26 h endurance with the framework fixes in place from a
   freshly-cleaned pool**, on whichever host's drive vendor we are most
   interested in characterising. Per-stream JSON percentiles will then be
   available for the primary endurance result, which we don't currently
   have for -08.

## 10. Open questions

- Why does -05 read at 60 % of -08's rate while writing 28 % faster on
  random small writes? Mixed asymmetry argues against a single cause.
- Why did the 26 h on -08 degrade monotonically while the 26 h on -05
  improved over the first 16 h then stabilised? Same workload, opposite
  drift shapes.
- Was there activity on -05 during testing that we don't have visibility
  into? Cross-reference `journalctl --since 2026-04-24` for VM start/stop,
  scrub activity, snapshot replication. None of that is in the fio result
  directories.
- Is the 36 ms p99 / 65 ms p99.9 SQL log commit latency on -05 reproducible
  on a freshly-cleaned pool, or is it specific to this pool's history?
- How much of the dataset-vs-zvol gap is virtio-vs-ZPL code path versus
  recordsize-vs-volblocksize alignment? A focused test using a *zvol with
  recordsize-equivalent volblocksize* could disambiguate.
- Does the IOPS-decreases-with-ARC trend persist or invert at long runtime?
  Hypothesis: at FIO_RUNTIME=1800 ARC populates and the trend reverses for
  read-bound work; for write-bound work like sql-log the dirty-data flush
  cost dominates and the trend may not invert. The re-run will tell us.
- On -08, does capping `zfs_dirty_data_max` independently of `arc_max`
  fix the sql-log regression at ARC=256 GiB? If yes, -08 has a
  per-host operational tuning (raise ARC, cap dirty data) that gives
  best-of-both-worlds. The dirty-data sweep is built for this.
- Why does -05 (Micron) handle large dirty-buffer flushes more gracefully
  than -08 (Samsung)? Controller architecture? Larger SLC cache? Different
  write-path scheduling? Worth correlating with NVMe spec sheets once
  the identity audit is done.
